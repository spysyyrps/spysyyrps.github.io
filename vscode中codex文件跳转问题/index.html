用AGENTS.md解决Codex文件链接跳转异常

本文最后更新于：2026年3月18日 下午

codex插件版本：26.5313.41514

vscode版本：1.111.0

操作系统：win11

在 VS Code 里使用 Codex 时，我遇到过一个很恼人的问题：当助手引用本地文件时，点击链接并不会直接打开对应文件，而是莫名其妙跳到浏览器，甚至出现类似 https://file+.vscode-resource.vscode-cdn.net/... 这样的地址。

这个问题看起来像是文件路径写错了，但实际上，问题并不在文件本身，而在扩展对链接的解析方式。后来我没有去直接修改扩展代码，而是采用了一种更轻量、也更稳定的规避方案：把一段规则写进 C://Users/username/.codex/AGENTS.md，强制 Codex 不再输出 Markdown 文件链接，而改用它原生可跳转的文件引用格式。

这篇文章就来完整记录这个问题，以及为什么这个方案有效。

问题现象

正常情况下，我们希望助手给出的文件引用能像这样工作：

  • 点击后直接打开本地文件
  • 最好还能定位到具体行号
  • 不要跳浏览器，不要走奇怪的网页地址

点击之后，VS Code 并没有把它当成本地文件打开，而是把它当成了一个普通网页链接处理，最后跳到了浏览器。

为什么会这样

简单来说，Codex 所在的聊天面板本质上是一个 WebView。

在这个 WebView 里，普通 Markdown 链接会先经过一层 URL 处理逻辑。如果这个链接没有被识别成“文件引用”，它就会退化成普通的网页 <a> 标签。

问题就在这里：

  1. Windows 本地路径并没有被稳定识别为“文件引用”
  2. 链接在中间转换过程中被当成普通 URL 处理
  3. 最终点击时走了浏览器打开逻辑，而不是 VS Code 的“打开文件”逻辑

所以表面上看是“链接坏了”，本质上其实是“链接类型被识别错了”。

解决思路

既然问题出在 Markdown 文件链接上，那最直接的规避办法就是：

不要再让 Codex 输出 Markdown 文件链接，而是强制它使用 Codex 自己可识别、可跳转的文件引用格式。

这样做有三个明显好处。

第一，它是持久规则。

不需要我每次对话都重复发一遍 prompt，也不需要临时提醒助手“不要用 Markdown 链接”。

第二，它是“低成本规避”，不是“高风险改扩展”。

如果直接改扩展前端逻辑，虽然能根治，但维护成本高，而且扩展升级后补丁可能失效。

这个方案的本质

需要说明的是，这不是根治方案，而是规避方案。

真正的根因仍然是扩展对本地文件链接的识别不够稳，尤其是在 Windows 路径和 Markdown 链接混用时，容易退化成普通浏览器链接。

而 AGENTS.md 的作用，是把输出格式约束住，避免走那条有问题的路径。

换句话说，它不是修好了扩展，而是避开了扩展最容易出错的那条分支。
